# RAGFlow 检索服务扩容测试报告

## 1. 背景

内网开发服务器部署 Dify + RAGFlow + EKB 知识库网关，目标支撑 **200 人使用 Dify chatflow**。负载模型：200 人 × 峰值 20% 活跃 × chatflow 单次并发 5 路检索 = **峰值 200 并发检索**。

> ⚠️ **负载模型是估算假设**，"20% 活跃""5 路并发"未从 Dify 日志反推验证。扩容前应从 Dify 访问日志统计真实并发分布。

## 2. 架构与检索链路

```
用户 → Dify chatflow → workflow 知识库节点
     → EKB 知识库网关 (uvicorn 4w, 8090) [缓存+熔断器]
     → RAGFlow 容器内 nginx (8088)
     → RAGFlow API (ragflow_server, Quart ASGI 单进程)
     → ES 检索 + Embedding 服务 (qwen3-vl-embedding-8b, 4096维)
```

| 组件 | 职责 | 规格 |
|------|------|------|
| EKB 知识库网关 | Dify External KB API 适配→RAGFlow；溯源/缓存/熔断/多KB拆分 | FastAPI v3.0, uvicorn 4w |
| RAGFlow API | 接收检索请求，调 embedding，查 ES | **Quart ASGI 单进程** |
| RAGFlow ES | chunk+向量存储检索 | 单节点 9G heap，62万 chunk |
| Embedding 服务 | query 向量化 | qwen3-vl-embedding-8b (VLLM) |

## 3. 问题背景：workflow 超时根因

`ragflow-server` 容器曾异常退出（ExitCode=255，非 OOM），导致检索链路断裂：Dify→网关(存活)→RAGFlow(挂)→连接失败重试3次→MaxRetriesExceededError→用户看到超时。

**修复**：重启容器 + 加 `--restart=unless-stopped`。

## 4. 关键技术发现（颠覆性）

### 4.1 RAGFlow 用 Quart（ASGI）不是 Flask
`api.apps:app` 是 `Quart`。之前给 `app.run()` 加的 `threaded=True` **完全无效**（Quart 忽略，日志 `threaded are not supported`）。"阶段一修复"是空操作，改善只是重启副作用。

### 4.2 检索路径已原生异步化
`Dealer.retrieval()`（`/ragflow/rag/nlp/search.py:573`）是 `async def`，embedding 和 ES 查询都走 `await thread_pool_exec(...)`（线程池 128 workers）。**不阻塞 asyncio 事件循环**。瓶颈不在"同步阻塞"。

### 4.3 真正瓶颈：单进程 ASGI 连接处理上限
单进程（Quart app.run 或 hypercorn 1w）在 50 并发时：accept 队列打满→`Connection refused`→容器内 nginx `upstream server temporarily disabled`→**502**→P95 飙到 6-7s。

### 4.4 各上游不是瓶颈
| 组件 | 压测结果 | 结论 |
|------|---------|------|
| Embedding 上游 | 100并发 P95 293ms，100%成功 | 非瓶颈 |
| ES 查询 | 0.05-0.6s | 非瓶颈 |
| MySQL | 151 max_connections，压测时3连接 | 非瓶颈 |
| ragflow-server CPU | 压测时7%（IO等待） | 非瓶颈 |

## 5. 全链路压测数据（经 EKB 网关 8090）

### 5.1 基线：单实例 Quart app.run（熔断器调宽后）
| 并发 | P50 | P95 | 成功率 | 主要失败 |
|------|-----|-----|--------|---------|
| 20 | 7ms | 1494ms | 12% | 503×88(冷启动) |
| 50 | 45ms | **7101ms** | 76% | 502×61 |
| 100 | 67ms | 203ms | 83% | 502/503 |
| 200 | 54ms | 117ms | 82% | 503×173 |

> ⚠️ **压测方法论局限**：失败请求（503/502 毫秒级）被计入 P95 拉低统计，"100/200并发P95 200ms"高估性能。严谨应**仅统计成功请求P95**。看成功率和50并发拐点更可信。真实有效并发约20-30。

### 5.2 方案对比（50并发拐点）
| 方案 | 50并发P95 | 成功率 | 评价 |
|------|-----------|--------|------|
| Quart app.run 单实例(基线) | 7101ms | 76% | 灾难 |
| **多实例 nginx LB(9380+9382)** | **89ms** | **94%** | ✅最优 |
| hypercorn 1 worker | 169ms | 36% | 单进程一样差 |
| hypercorn 2 workers | 4754ms | 87% | 中等并发塌陷 |

### 5.3 多实例 nginx LB 完整数据（2实例）
| 并发 | P50 | P95 | 成功率 |
|------|-----|-----|--------|
| 20 | 984ms | 1447ms | 99% |
| 50 | 33ms | **89ms** | 94% |
| 100 | 69ms | 147ms | 95% |
| 200 | 62ms | 115ms | 95% |

**2实例把50并发P95从7101ms降到89ms（80倍提升）**，成功率76%→94%。

## 6. 方案审核结论

- **方案A（threaded=True持久化）**：❌无效。Quart忽略threaded。
- **方案B（多实例+nginx LB）**：✅最优，已验证。update_progress幂等安全，chat_channels空跑无害。风险：MySQL连接数。
- **方案C（gevent monkey patch）**：❌无效。Quart是asyncio不是gevent。
- **方案D（调网关熔断器）**：✅已做止血。`fail_threshold 5→20, cooldown 30→10, half_open 1→3`。暴露真实瓶颈（502）。
- **方案E（hypercorn多worker）**：⚠️不如多实例。50并发P95 4754ms塌陷。

## 8. 遗留问题
1. start_fastapi.sh PG/Redis IP已改动态获取（原写死172.22.0.3/0.4会过时）✅
2. 旧start.sh已标记废弃 ✅
3. 直连ragflow高并发401：走proxy无此问题，直连20+并发大量401（token校验并发失败，疑似DB查token超时）。**潜在安全风险**。推荐缓解：短期走nginx不直连，长期RAGFlow侧token缓存(Redis TTL 60s)。
4. ragflow-server crash根因未定位：ExitCode=255非OOM，日志被/api/v1/version健康检查噪音淹没。
5. /api/v1/version健康检查噪音：需定位并停掉探测脚本。
6. 多实例自动持久化：systemd watcher方案实测不稳定（见第10节），当前需容器重启后手动跑start-multi.sh。
7. MySQL max_connections=151：起4+实例需先调大。

## 9. 扩容建议（最终）—分阶段、有条件

> ⚠️ **推荐分阶段**（对抗辩论修正：原"直接起4实例"与"crash同因爆炸半径"自相矛盾）

**阶段1（立即低风险）**：固化2实例配置
1. 持久化2实例+nginx upstream（start-multi.sh）
2. EKB网关不变（仍指8088）
3. 加crash监控

**阶段2（满足前置条件后）**：扩到4实例
- 前置：crash根因定位 + 真实并发从Dify日志验证 + 阶段1稳定1周
- 扩到4实例+nginx upstream，调大MySQL max_connections

**为什么不直接上4实例**：crash根因未定位→4进程可能同因同时挂（爆炸半径）；2→4外推未验证；2实例已解当前痛点。

### 预期容量
- 2实例：50并发P95 89ms，成功率94%（**已实测**）
- 4实例：预期100-200并发P95<200ms（⚠️待验证假设，89ms已近embedding+ES理论下限，不会线性提升）

### 风险：多实例爆炸半径
- crash同因风险：4实例可能同时挂。需定位crash根因后再扩。
- nginx单点：多实例后容器内nginx成新单点，是否瓶颈未测。

### 验收标准
- 阶段1：50并发P95<200ms成功率>90%（已达标89ms/94%）、真实流量1周无crash、MySQL连接<100
- 阶段2：200并发P95<500ms成功率>90%、4实例独立crash不影响其他、nginx不成瓶颈

## 10. 持久化方案探索与教训

### 10.1 改 entrypoint 方案（未采用）
- 优点：容器启动自动配，一劳永逸
- 缺点：RAGFlow升级时官方entrypoint变更需手动合并；需重建容器中断3-5分钟；维护成本高
- 已有custom-entrypoint.sh但未被deploy.sh启用（之前有人试过放弃，可能因维护成本）

### 10.2 systemd watcher 方案（实测不稳定，已禁用）
写了`multi-watcher.sh` + `ragflow-multi-watcher.service`（user-level systemd），监听docker events自动跑start-multi.sh。
- **问题1**：`docker events`模板语法`{{.Status}}`在当前docker版本不存在，改`{{.Action}}`
- **问题2**：`docker events`在systemd user环境立即退出（每10s crash-loop，systemctl Restart=always循环重启）
- **问题3**：容器restart触发后，watcher在等待9380就绪期间未完成配置（9382没起、nginx没配）
- 改轮询版本仍crash-loop（systemd主动Stopping，原因未定位，疑似环境/PATH/pipefail）
- **结论**：systemd+docker events方案不可靠。已禁用清理。当前接受"容器重启后手动跑start-multi.sh"，后续探索更可靠方案（如docker healthcheck+sidecar，或改entrypoint）。

### 10.3 当前持久化状态
- start-multi.sh已就绪（幂等，手动跑可靠）
- 容器重启后需手动：`bash /home/ai.hse/ragflow-deploy-v2/start-multi.sh 2`
- start_fastapi.sh已固化动态IP+熔断器env（proxy重启可靠）

## 11. 对抗辩论结论（adversarial-spec）

本报告经两轮对抗审查（glm-5.2 + deepseek-v4-pro + Claude自评；kimi-k3超时）。

**Round 1采纳**：方法论局限、外推标注、负载假设、爆炸半径、401缓解方向、custom-entrypoint矛盾。

**Round 2采纳**（修正R1自相矛盾）：推荐改分阶段有条件、加验收标准、401给推荐方案。

**preserve-intent保留**：多实例最优、hypercorn否定、Quart vs Flask发现（均有实测支撑）。

**未决**：crash根因、401根因、4实例实测、真实并发分布、systemd自动持久化。

---

**核心结论**：RAGFlow检索瓶颈是**单进程ASGI连接处理上限**。最优解是**多实例+nginx负载均衡**——2实例已把50并发P95从7101ms降到89ms。**分阶段推进**：阶段1固化2实例，阶段2满足前置条件后扩4实例。
