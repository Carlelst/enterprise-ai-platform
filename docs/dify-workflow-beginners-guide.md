# Dify 工作流搭建入门指南

> 适用版本：Dify 1.x+ | 难度：入门 | 预计阅读：20 分钟

---

## 一、什么是 Dify 工作流

Dify 是一个开源的 LLM 应用开发平台。**工作流（Workflow）** 是 Dify 的核心能力之一——你可以像搭积木一样，把大模型（LLM）、知识库（RAG）、代码逻辑、API 调用等组件拖拽连接起来，构建出能解决实际问题的 AI 应用。

**一句话理解**：工作流 = 可视化编排的 AI 应用流水线。

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  开始节点  │───▶│  LLM 节点  │───▶│ 条件判断  │───▶│  结束节点  │
│ (输入问题) │    │ (AI 推理) │    │ (分支路由) │    │ (输出结果) │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```

---

## 二、两类应用：Chatflow vs Workflow

Dify 提供两种工作流形态，选型对比：

| 维度 | Chatflow（对话流） | Workflow（标准工作流） |
|------|-------------------|----------------------|
| **交互模式** | 多轮对话，有记忆 | 单次执行，有输入→有输出 |
| **适用场景** | 客服机器人、AI 助手 | 数据处理、批量任务、自动化 |
| **输出方式** | Answer 节点流式回复 | End 节点返回结构化结果 |
| **变量管理** | 支持会话变量 | 仅工作流内变量 |
| **典型例子** | "帮我查一下订单状态" | "把 CSV 数据做汇总分析" |

**选型建议**：需要和人聊天选 Chatflow；需要自动跑任务选 Workflow。

## 三、快速入门：10 分钟搭建第一个工作流

### 3.1 准备工作

1. 确保你有 Dify 账号（自部署或 Dify Cloud）
2. 配置好至少一个 LLM 模型提供商（如 OpenAI API Key）
3. 进入 Dify 首页，点击 **"创建应用" → 选择 "工作流"**

### 3.2 Hello World 工作流：智能翻译助手

让我们搭建一个最简单的翻译工作流，感受完整流程。

**第 1 步：设置开始节点**

点击画布上的 **开始（Start）** 节点，添加输入变量：

| 变量名 | 类型 | 说明 |
|--------|------|------|
| `text` | Text | 待翻译的文本 |
| `target_lang` | Text | 目标语言（如"英文"、"日文"） |

**第 2 步：添加 LLM 节点**

从左侧面板拖拽 **LLM** 节点到画布，连线从开始节点接过来。

配置：
- **模型**：选择 GPT-4 或 Claude
- **系统提示词**：
  ```
  你是一个专业翻译。请将用户输入的文本翻译成{{target_lang}}。
  只返回翻译结果，不要解释。
  ```
- **用户提示词**：
  ```
  {{start.text}}
  ```

> 💡 `{{变量名}}` 是 Dify 的变量引用语法，会自动替换成实际值。

**第 3 步：添加结束节点**

LLM 节点后面默认连着 **结束（End）** 节点。在 End 节点中设置输出变量：选择 LLM 节点的 `text` 输出。

**第 4 步：测试**

点击右上角 **"运行"** 按钮，输入测试数据：
- `text`: "你好世界"
- `target_lang`: "英文"

点击执行，你应该看到输出：`Hello world`

**第 5 步：发布**

点击右上角 **"发布"**，你的翻译助手就上线了！Dify 会自动生成 API 端点，可以集成到其他应用中。

## 四、核心概念

### 4.1 节点（Node）

节点是工作流的最小功能单元。Dify 提供了 **20+ 种节点**，分为六大类：

| 类别 | 包含节点 | 作用 |
|------|---------|------|
| 🚪 **入口** | Start、Webhook 触发器、定时触发器 | 定义输入的来源和方式 |
| 🧠 **AI** | LLM、知识检索、问题分类、参数提取、Agent | 调用大模型和知识库能力 |
| 🔀 **逻辑** | 条件分支(IF/ELSE)、迭代(Iteration)、循环(Loop) | 控制流程走向 |
| 🔧 **数据处理** | Code、模板转换、变量聚合、变量赋值、列表操作、文档提取 | 转换和加工数据 |
| 🌐 **外部集成** | HTTP 请求、工具调用 | 调用外部 API 和服务 |
| 📤 **输出** | Answer、End | 产生最终结果 |

### 4.2 变量（Variable）

变量是数据在工作流中流动的载体。Dify 的变量系统分层设计：

| 变量类型 | 作用域 | 说明 |
|----------|--------|------|
| **节点输出变量** | 下游节点 | 每个节点执行后自动产生的输出，如 LLM 节点的 `text` |
| **开始节点变量** | 全局 | 用户输入，整个工作流都能引用 |
| **循环变量** (Loop) | 循环体 | 跨迭代累积的数据，每次循环都可以读取和更新 |
| **会话变量** (Chatflow) | 对话生命周期 | 跨多轮对话持久化 |

**变量引用语法**：
```jinja2
{{节点名.输出字段}}          # 如 {{llm.text}}
{{start.user_input}}        # 引用开始节点的输入
```

**变量赋值操作**（Variable Assigner 节点）：
- 覆盖、清除、追加、算术运算（加减乘除）

### 4.3 连线（Edge）

节点之间的箭头连线决定了执行顺序。本质上是一个**有向无环图（DAG）**：

- 数据从上游节点流向下游节点
- 分支节点（如 IF/ELSE）产生多条路径
- 聚合节点（如 Variable Aggregator）合并多条路径

### 4.4 上下文（Context）

Dify 的工作流上下文：
- **模型上下文**：LLM 节点的系统提示词 + 用户提示词 + 对话历史（Chatflow 中）
- **变量上下文**：当前节点能访问的所有上游变量
- **知识库上下文**：知识检索节点从文档中召回的相关片段

## 五、节点类型详解

### 5.1 入口节点

#### Start（开始节点）

一切工作流的起点。在这里定义用户需要提供哪些输入。

| 字段类型 | 说明 |
|----------|------|
| Text | 短文本输入 |
| Paragraph | 长文本（多行） |
| Number | 数字 |
| Select | 下拉选择 |
| File | 文件上传 |
| File List | 多文件上传 |

**最佳实践**：变量名用英文（如 `user_query`），便于在代码节点中引用。

#### Webhook 触发器

通过 HTTP 请求触发工作流执行。适合将工作流作为 API 服务暴露给外部系统调用。

```bash
curl -X POST https://your-dify.com/v1/workflows/run \
  -H "Authorization: Bearer <api-key>" \
  -H "Content-Type: application/json" \
  -d '{"inputs": {"text": "你好世界"}}'
```

---

### 5.2 AI 节点

#### LLM 节点 —— 最核心的节点

调用大语言模型进行推理和文本生成。

**关键配置：**

| 配置项 | 说明 |
|--------|------|
| 模型选择 | 选哪个 LLM（GPT-4、Claude、qwen 等） |
| 系统提示词 | 设置 AI 的角色和行为规则 |
| 用户提示词 | 用户的具体任务（可引用变量） |
| 记忆窗口 | Chatflow 中保留多少轮对话历史 |
| 结构化输出 | 强制 LLM 按 JSON Schema 输出 |

**示例 —— 结构化输出**：

在 LLM 节点中启用"结构化输出"，定义 JSON Schema：
```json
{
  "sentiment": "positive|negative|neutral",
  "keywords": ["关键词1", "关键词2"],
  "summary": "一句话总结"
}
```

这样 LLM 就会严格按照这个格式输出，下游节点可以可靠地解析。

#### 知识检索节点（RAG）

从你的知识库中检索相关文档片段，送入 LLM 做增强生成。

**配置步骤**：
1. 选择知识库（需要提前在"知识库"模块上传文档）
2. 设置检索策略（向量检索 / 全文检索 / 混合检索）
3. 设置召回数量（Top-K）
4. 设置相似度阈值

#### 问题分类节点

用 LLM 做意图识别，将用户输入路由到不同分支。

```
用户输入 → 问题分类 → ├─ 类别A（咨询）→ 知识库查询
                      ├─ 类别B（投诉）→ 人工处理
                      └─ 类别C（闲聊）→ LLM 闲聊
```

#### 参数提取节点

从自然语言中提取结构化信息。例如从"我要订一张明天北京到上海的机票"中提取：

```json
{"departure": "北京", "destination": "上海", "date": "2026-06-30"}
```

#### Agent 节点 —— 最强大的节点

Agent 节点让 LLM 能够**自主选择和使用工具**。它采用 ReAct（推理-行动）策略：
1. 分析任务，决定需要什么工具
2. 调用工具获取结果
3. 根据结果继续推理或调用其他工具
4. 循环直到完成任务

**配置工具**：可以配备搜索（Google/Exa）、代码执行、知识库查询、自定义 API 等多种工具。

**⚠️ Agent 注意事项**：
- 比普通 LLM 节点消耗更多 Token（每次工具调用都要请求 LLM）
- 设置最大推理轮次防止无限循环（建议 5-10 轮）
- 只传递必要信息给 Agent，不要灌入全部上下文

---

### 5.3 逻辑控制节点

#### IF/ELSE 条件分支

根据条件决定流程走向。

**支持的条件操作符**：
- 等于、不等于、包含、不包含
- 大于、小于、大于等于、小于等于
- 为空、不为空
- 支持 AND / OR 组合多个条件

```
开始 → IF 判断
       ├─ 条件成立（temperature > 30）→ "天气炎热，建议开空调"
       └─ 条件不成立（temperature ≤ 30）→ "天气适中"
```

#### Iteration 迭代节点

对数组中的每个元素执行相同的处理逻辑。

**典型场景**：
- 批量处理：对一组图片逐一分析
- 多步骤：对列表中的每一项执行 LLM → Code → HTTP 子流程

```
输入：["苹果", "香蕉", "橙子"]
迭代节点 → 对每个水果执行 → LLM 翻译 → 收集结果
输出：["Apple", "Banana", "Orange"]
```

#### Loop 循环节点

重复执行直到满足退出条件。

**核心机制**：
- 设置最大循环次数（防止死循环）
- 使用**循环变量**在迭代间传递状态
- 设置**退出条件**提前终止

**区别于 Iteration**：Loop 用于"未知次数"的循环（如搜索直到找到足够信息），Iteration 用于"已知长度"的遍历。

---

### 5.4 数据处理节点

#### Code 节点

在 Python 或 JavaScript 中写自定义逻辑。

**Python 示例 —— 计算两个数字之和**：
```python
def main(arg1: int, arg2: int) -> dict:
    result = arg1 + arg2
    return {
        "sum": result,
        "is_even": result % 2 == 0
    }
```

**使用场景**：
- 数据格式转换（JSON ↔ 文本）
- 数学计算
- 条件逻辑（比 IF/ELSE 更灵活）
- 时间日期处理

#### 模板转换节点

用 Jinja2 模板将变量组合成格式化文本。

```jinja2
查询结果摘要：
- 查询词：{{start.query}}
- 召回文档数：{{retrieval.docs | length}}
- 内容片段：
  {% for doc in retrieval.docs %}
  * {{doc.title}}: {{doc.summary}}
  {% endfor %}
```

#### 变量聚合节点

当多条分支汇合时，聚合不同分支产生的变量。Dify v0.6.10+ 支持多聚合组，可以将不同分支的输出分区管理。

#### 列表操作节点

对数组进行过滤、排序、截取操作，无需写代码。

| 操作 | 说明 |
|------|------|
| Filter | 按条件筛选（如 `{{item.score}} > 0.8`） |
| Sort | 按字段排序 |
| First/Last N | 取前/后 N 条 |

#### 文档提取节点

解析上传的 PDF、Word、TXT 文件，提取纯文本内容，送入下游处理。

---

### 5.5 外部集成节点

#### HTTP 请求节点

调用任意外部 REST API。

**配置示例 —— 调用天气 API**：
```
方法：GET
URL：https://api.weather.com/v1/current?city={{start.city}}
Headers：Authorization: Bearer {{env.WEATHER_API_KEY}}
```

**使用技巧**：
- 敏感信息（API Key）存在环境变量中，用 `{{env.KEY_NAME}}` 引用
- 请求体中使用变量：Body → JSON → `{"query": "{{start.question}}"}`

#### 工具调用节点

调用 Dify 内置的 40+ 工具，包括：
- 搜索引擎：Google、Bing、Exa
- 办公协作：Notion、GitHub、Jira
- 图像处理：Stable Diffusion、DALL·E
- 自定义工具：通过 API 工具导入

---

### 5.6 输出节点

#### End 节点（Workflow）

定义工作流的最终输出。可以输出多个变量（JSON 格式）。

```
输出变量：
  - result: {{llm.text}}
  - source: {{retrieval.docs}}
  - timestamp: {{sys.timestamp}}
```

#### Answer 节点（Chatflow）

流式输出聊天回复，支持 Markdown 格式、图片嵌入和引用标注。

## 六、实战案例

### 案例 1：智能 FAQ 客服机器人

**需求**：用户提问→查知识库→有答案直接回；不确定性高→让用户确认后回。

**工作流设计**：

```
开始（用户问题）
  │
  ▼
知识检索（查 FAQ 知识库）
  │
  ▼
IF 判断（retrieval.score > 0.8）
  ├─ 是（高置信度）
  │   └─ LLM 整理答案 → Answer 节点 → 结束
  │
  └─ 否（低置信度）
      └─ LLM 生成反问（确认用户意图）
          → Answer 节点（请用户确认）
          → 结束
```

**关键配置**：

1. **知识检索**：上传 FAQ 文档到知识库，设置 Top-K=3，score 阈值=0.7
2. **LLM 整理答案**：提示词"根据以下知识库内容回答用户问题：{{retrieval.output}}"
3. **LLM 反问**：提示词"以下问题无法从知识库找到答案，请生成3个确认性问题帮用户澄清意图"

### 案例 2：自动化数据分析工作流

**需求**：用户上传 CSV 文件→自动分析→生成可视化报告。

**工作流设计**：

```
开始（上传 CSV）
  │
  ├─ 字段1：file（File 类型）
  ├─ 字段2：analysis_type（Select：汇总/趋势/对比）
  │
  ▼
文档提取（解析 CSV 内容）
  │
  ▼
Code 节点（Python：Pandas 数据处理）
  │ → 计算统计指标（均值、中位数、标准差等）
  │ → 输出结构化 JSON
  │
  ▼
IF 判断（analysis_type）
  ├─ 汇总 → LLM（生成汇总报告）
  ├─ 趋势 → LLM（生成趋势分析）
  └─ 对比 → LLM（生成对比报告）
  │
  ▼
模板转换（格式化为 Markdown 报告）
  │
  ▼
End 节点（输出报告）
```

**关键 Code 节点代码**：
```python
import pandas as pd
import json

def main(csv_text: str, analysis_type: str) -> dict:
    import io
    df = pd.read_csv(io.StringIO(csv_text))
    
    stats = {
        "row_count": len(df),
        "columns": list(df.columns),
        "numeric_summary": {}
    }
    
    for col in df.select_dtypes(include=['number']).columns:
        stats["numeric_summary"][col] = {
            "mean": round(df[col].mean(), 2),
            "median": round(df[col].median(), 2),
            "max": round(df[col].max(), 2),
            "min": round(df[col].min(), 2)
        }
    
    return {"stats": json.dumps(stats, ensure_ascii=False)}
```

### 案例 3：智能搜索 + 深度研究工作流

**需求**：给定研究主题→自动搜索网络→多轮深入→生成带引用的研究报告。

**工作流设计**：

```
开始（research_topic + max_loops）
  │
  ▼
LLM（意图分析：拆解研究子问题）
  │
  ▼
Loop（迭代变量：findings、visited_urls、knowledge_gaps）
  │
  ├─ LLM（生成搜索方向 + 搜索关键词）
  │
  ├─ Agent（配备搜索工具，执行网络搜索）
  │
  ├─ Code（提取 URL 和引用信息）
  │
  ├─ 变量赋值（更新 findings、visited_urls）
  │
  └─ 退出条件：knowledge_gaps 为空 OR 达到 max_loops
  │
  ▼
LLM（综合所有 findings，生成研究报告）
  │
  ▼
End（输出报告 + 引用列表）
```

**提示词设计**（综合报告 LLM）：
```
你是一个研究分析专家。请综合以下研究发现的碎片，生成结构化的研究报告。

研究发现：
{{loop.findings}}

要求：
1. 有清晰的摘要、分章节论述、结论
2. 包含引用标记 [1] [2] 等
3. 使用 Markdown 格式

参考资料：
{{loop.visited_urls}}
```

## 七、最佳实践

### 7.1 设计原则

| 原则 | 说明 |
|------|------|
| **职责单一** | 每个节点只做一件事，不要在一个 LLM 节点里塞太多任务 |
| **分阶段设计** | 把复杂任务拆成意图识别→执行→校验→输出多个阶段 |
| **失败友好** | 关键步骤加错误处理和降级逻辑 |
| **先简后繁** | 先用最少节点跑通，再逐步加分支和优化 |

### 7.2 变量管理

- 变量名用英文，语义清晰（`search_results` 不是 `sr`）
- 善用**变量赋值节点**管理复杂状态（特别是 Loop 中）
- 环境变量存敏感信息：`{{env.API_KEY}}`
- 下游只引用必需的变量，不要"全透传"

### 7.3 LLM 节点优化

```yaml
# ❌ 不好的提示词
"帮我分析这个数据"  # 太模糊

# ✅ 好的提示词
"""
你是一个数据分析师。请分析以下数据：
数据：{{code.stats}}
分析维度：
1. 数据整体概览（2-3句话）
2. 关键发现（最多3条）
3. 数据质量评估（是否有缺失值、异常值）
输出为 Markdown 格式。
"""
```

**关键技巧**：
- 指定角色、任务、格式
- 用 `{{变量}}` 精确注入数据
- 数量限制（"最多3条"、"2-3句话"）
- 启用结构化输出确保格式一致

### 7.4 错误处理

Dify 工作流中的错误处理策略：

| 场景 | 策略 |
|------|------|
| **LLM 调用失败** | 设置重试（模型节点自带重试选项） |
| **HTTP 请求超时** | 设置超时时间 + 降级返回默认值 |
| **知识库召回为空** | IF 判断 → 降级为 LLM 通用回答 |
| **Agent 跑偏** | 设置 max_iterations，提示词加约束 |

**IF/ELSE 作降级模式**：
```
HTTP 请求节点
  │
  ▼
IF（HTTP.status_code == 200）
  ├─ 成功 → 继续处理
  └─ 失败 → LLM（"API 调用失败，请用您自己的知识回答"）→ 继续
```

### 7.5 性能优化

- **并行执行**：无依赖的节点会并行运行，缩短总延迟
- **减少 LLM 调用**：能用 Code 节点解决的不要用 LLM
- **缓存检索**：知识库预热（提前索引），减少检索耗时
- **小模型分流**：简单分类/提取任务用 qwen-turbo 或 haiku，复杂推理再用大模型

### 7.6 版本管理与迭代

- 每次重大修改前，先 **"另存为新版本"**
- 发布前在测试环境跑完所有分支
- 善用工作流的 **运行日志**（Logs）排查问题
- 生产环境用环境变量切换配置（如 API 地址）

## 八、调试与测试

### 8.1 单步调试

在画布中点击任一节点的 **"运行此节点"**（播放图标），可以单独测试该节点：
1. 手动输入上游变量值
2. 执行并查看输出
3. 检查是否符合预期

### 8.2 全流程测试

1. 点击右上角 **"运行"**
2. 填入开始节点的输入变量
3. 观察每一步的执行结果和耗时
4. 查看节点输出面板，确认数据流正确

### 8.3 日志分析

发布后，在应用页面的 **"日志"** 标签中可以查看：
- 每次工作流执行的完整链路
- 每个节点的输入/输出
- Agent 节点的推理日志和工具调用记录
- 执行耗时和 Token 消耗

### 8.4 常见问题排查

| 问题 | 原因 | 解决 |
|------|------|------|
| LLM 输出格式不稳定 | 未使用结构化输出 | 启用"结构化输出"定义 JSON Schema |
| 知识库检索无结果 | 阈值太高或文档不匹配 | 降低阈值，检查文档覆盖 |
| Loop 无限循环 | 退出条件永远不满足 | 检查退出条件逻辑 + 设置 max_loops |
| Code 节点报错 | Python 依赖缺失或语法错误 | 查看错误日志，Dify Code 节点仅支持标准库 |
| 变量引用报 "not found" | 节点名或变量名拼写错误 | 使用变量选择器（点选而非手输） |

## 九、部署与发布

### 9.1 发布应用

1. 在画布页顶部点击 **"发布"**
2. 发布后应用进入**只读模式**，不可编辑
3. 可在应用概览页获取 API 密钥

### 9.2 API 调用

```bash
# Workflow 调用
curl -X POST https://your-dify.com/v1/workflows/run \
  -H "Authorization: Bearer app-xxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": {"query": "Dify 如何部署"},
    "response_mode": "blocking",
    "user": "user-123"
  }'

# Chatflow 调用（流式）
curl -X POST https://your-dify.com/v1/chat-messages \
  -H "Authorization: Bearer app-xxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": {},
    "query": "你好",
    "response_mode": "streaming",
    "user": "user-123"
  }'
```

### 9.3 嵌入前端

Dify 支持将应用嵌入网页，有两种方式：
1. **iframe 嵌入**：复制嵌入代码，粘贴到网站 HTML 中
2. **SDK 集成**：使用 Dify SDK 在自定义前端中调用 API

### 9.4 监控与迭代

- 定期查看 **"日志"** 页面，了解用户使用情况
- 关注**失败率和 Token 消耗**
- 根据用户反馈和日志数据持续优化提示词和流程

  ├─ 添加错误处理和降级逻辑
  ├─ 优化提示词和性能
